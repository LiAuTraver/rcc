##### Conventions

Appearently conventions below are C++-ish and probably considered _bad practice_ by Rust community.

###### module

- `fmt`: implementation for `std::fmt::*`. private. Use `fmt::*` rather than directly import trait, since there's no ADL, `.fmt(f)` method can conflict with each other.
- `cvt`: implementation of `std::convert::From`, etc. private.
- `fwd`: re-export `external` crate to enable transitive dependency. Bad practive as said, but for historical reason. private.
- `::`: purely for historic reason, now for style consistency. prefix any `extern`al crate with global specs `::`, like `::std::ptr::eq` rather than `std::ptr::eq`.
- prefer `use A as B` than `type B = A`.
- use `use <enum>::*` sparingly in global scope.
- `use come_crate::*` is not allowed. pls spell it out what's required instead of wildcard.
- `super::super` is discouraged.

###### struct

- private struct name can be short, like `Span`; however public re-export(if any) should be descriptive, like `SourceSpan`.
- `new` as ctor. even for aggregate, since structured bindings applies to public fields and irrelavent w/ ctor.
- member field should stay private if it's not a POD type. Don't write java, nonetheless. Also, be consistent. _Partially_ public field is ugly and misleading. Either full `pub` as POD or full getter.
- `Deref` used as public inheritance, or dirty way to save typing.
- `dyn`/_dynamic dispatch_ is not allowed as for now. So does `Any`. No more `dynamic_cast` when we have built-in `enum`! Manually rolling out static dispatch or use the util macro.
- non-`consteval` static global variable is forbidden.
- `#[derive(Copy)]` based on the functionality of the type, not based on whether it _needs Drop_.
- prefer use `Enum::Variant()` than `Enum::Variant` if the `Enum` is not strictly a `unit enum` but a discriminated union.
- do not use `Enum::Variant{field1: type1, field2: type2}` or `Enum::Variant(field1: type1, field2: type2)`. it's hard to nevigate as well as difficult to `match`. use `Enum::Variant(Variant)` and `struct Variant` instead.
- make meaningfule types, not tuples, in the exception of temporary function return type.
- prefer splitting `impl` section for clarity, like `#pragme region`.
- `#[allow(non_upper_case_globals)]` for static constexpr class field.

###### function

- spam `#[must_use]`.
- make function as `constexpr` as possible.
- `inline` getter and setter as much as possible, I prefer `[[gnu::always_inline]]`.
- create stray function sparingly. it's hard to conjure up from memory. This does not mean it's forbidden; we are not writting java. static member function is perferred if the function serves the suitable purpose.
- prefer closure than static member function if the relavent operation is only used in current scope. sometimes borrowck conplains, use local function instead.
- prefer trait-based function overloading, like `trait DoThing` than mangling function name manually by typing `fn do_thing_for_...` if the target is algeberic data type, aka, discriminated union, and has many variants.
- use operator overloading cleverly. We are not writting java.
- make template when the param is always a `constexpr`.
